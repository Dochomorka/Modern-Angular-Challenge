# Day 40 — Capstone Project and Review

Day 40 is the final review and capstone day. Angular’s docs describe Angular as a framework for building scalable web applications, which makes this the right point to bring together components, forms, testing, performance, accessibility, and deployment into one finished project [web:412][web:416][web:379].

## Goal

By the end of this day, you should be able to:

- Combine what you learned into one project.
- Review your strongest and weakest topics.
- Explain your architectural choices.
- Identify remaining gaps.
- Present a working Angular app.
- Leave with a clear next-step plan.

## Why This Matters

A capstone project proves that you can connect isolated skills into a working app. Angular’s overview and docs emphasize building efficient and scalable applications, and the final review helps confirm that your code is usable, maintainable, and production-ready [web:412][web:416][web:379].

This matters because:
- Real projects need multiple skills together.
- Review helps fix weak spots before shipping.
- A capstone shows your actual progress.
- It gives you something portfolio-ready.

## Capstone Scope

A good final project should include a few major parts from the roadmap:
- Components and routing.
- Forms and validation.
- State management.
- Service interaction.
- Testing.
- Performance and accessibility basics.
- Deployment-ready structure.

The exact scope can stay small, as long as it demonstrates the core skills clearly [web:379][web:384][web:386].

## Example Project Ideas

A capstone can be simple but complete. Good examples include:
- Task manager.
- Contact manager.
- Mini dashboard.
- Expense tracker.
- Recipe app.

Each one can show forms, state, services, and testing without becoming too large.

## Review Checklist

Use this final review to check:
- Can the app load and navigate correctly?
- Are forms validated properly?
- Is state shared cleanly?
- Are components reusable?
- Do tests pass?
- Is the app accessible?
- Is the build ready for deployment?

## Practical Example

```md
## Capstone: Task Manager

Features:
- Add tasks through a form.
- Validate task input.
- Mark tasks complete.
- Store tasks in a service.
- Reuse a task card component.
- Test one service and one component.
- Build for production.
- Verify keyboard and label accessibility.
```

This kind of project uses the same Angular patterns covered across the roadmap, especially component composition, forms, service-based state, and testing [web:386][web:340][web:359][web:351].

## Best Practices

- Keep the capstone focused and realistic.
- Use the simplest architecture that works.
- Reuse patterns from earlier days.
- Test the most important behavior.
- Fix accessibility before calling it done.
- Build and preview the app before finalizing.

## Easy Challenges

- Pick one capstone idea.
- List the features it needs.
- Reuse one shared component.
- Add one form with validation.
- Test one important screen.

## Medium Challenges

- Build the main workflow end to end.
- Add a shared service for state.
- Create one reusable UI piece.
- Add at least two tests.
- Run a production build.

## Hard Challenges

- Finish the full app flow.
- Polish accessibility.
- Reduce repeated logic.
- Improve structure and naming.
- Review and refactor weak areas.

## Reflection Questions

- What did you learn most clearly?
- Which topic still feels weak?
- What architecture choices worked best?
- What would you change in a second version?
- What would make this project portfolio-ready?

## Final Deliverable

Finish one small Angular app that includes:

- At least one form.
- At least one shared state or service.
- At least one reusable component.
- At least one test.
- A production build.
- Basic accessibility checks.

## Final Review Flow

1. Pick the project.
2. Build the core workflow.
3. Add form validation.
4. Add service/state logic.
5. Add tests.
6. Run a production build.
7. Review accessibility and structure.

## Closing Note

This is the end of the roadmap, but it is also the start of using Angular in real projects. The goal now is not just to know the topics, but to use them together with confidence [web:412][web:416].
